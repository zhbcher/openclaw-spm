# Enhanced Phase 1 — 深度需求访谈 (Prometheus 模式)

> 借鉴 OMO Prometheus 规划师。在 Soul-Searching 之前先进行意图分类 + 研究前置 + 测试评估 + 范围锁定。

## 触发条件

- SPM Phase 1 启动时（替代原有 Soul-Searching Protocol 的默认路径）
- 用户说"帮我规划""帮我设计""我想要做"等需求不明确的启动语句
- **例外**：用户已给出极具体的 spec（精确到文件/行/API 签名）→ 跳过，直接用 Brainstorming

---

## Step 1: 意图分类（必须先做）

**在问任何问题之前，先判定任务类型。分类决定了后续所有行为。**

| 意图类型 | 触发词 | 策略 |
|---------|--------|------|
| **轻量修改** | "修一下""改个""替换"、单文件、<10行 | 快速确认→跳过深度访谈 |
| **重构** | "重构""整理""拆分""迁移" | 安全优先：先扫影响范围再问 |
| **新建** | "新建""创建""实现"、新模块/项目 | 探索优先：先扫代码库模式再问 |
| **中型任务** | 多文件、有明确范围 | 边界优先：锁定确切产出 |
| **架构决策** | "架构""选型""设计" | 战略优先：必须派审查子代理 |
| **探索研究** | "调研""看看有没有""评估" | 调研优先：并行探测+退出条件 |

### 轻量修改 → 快速通道

跳过深度访谈，直接：
1. 确认理解："我理解你要做 X，对吗？"
2. 追问 1 个关键问题（不展开）
3. 提议行动 → 等确认 → 进入 Phase 2

---

## Step 2: 研究前置（先扫再问）

**在向用户提问前，先收集事实。不问"你的代码库里有没有 XX"，而是直接扫出来告诉用户。**

```
执行以下扫描（并行）：

1. 代码库模式扫描
   → 找 2-3 个最相似的已有实现
   → 提取：目录结构、命名规范、导出方式、错误处理模式
   → 用途：照搬约定，不引入新风格

2. 依赖与限制扫描
   → package.json / pyproject.toml 中的已有依赖
   → 是否已有类似库（防重复引入）
   → 语言版本、框架版本约束

3. 配置与规范扫描
   → eslint/prettier/tsconfig 配置
   → 已有的 AGENTS.md / CLAUDE.md 中的硬性约定
   → 项目特有的命令（npm scripts、Makefile）

4.（仅新建/架构）外部方案扫描
   → 主流方案对比（官方文档 + 2-3 个生产级开源实现）
   → 不读教程、不看入门指引
```

**输出格式**（直接呈现给用户，不等用户问）：

```
扫描结果摘要：
- 已有类似实现：src/routes/users.ts（12 个接口，Express Router 模式）
- 已有依赖：express@4.18, zod@3.23
- 项目约定：文件命名 kebab-case, 接口名前缀 I 禁止使用

基于以上，我的建议是：[一句话方向]
确认后我会问几个问题确认细节。
```

---

## Step 3: 测试基建评估（新建/重构必须）

```
检测：
- 测试框架：□ Jest □ Vitest □ pytest □ 无测试
- 测试覆盖：□ 有配置 □ 无
- CI 中的测试命令：□ 有 □ 无

根据检测结果提问：

[有测试基建] → "你的项目有 Vitest。这次工作要包含自动化测试吗？
  - TDD（RED-GREEN-REFACTOR）
  - 实现后再补测试
  - 不加测试"

[无测试基建] → "项目目前没有测试框架。要顺便搭建吗？
  - 搭建 [推荐框架] + 示例用例
  - 先不加，本次只实现功能"
```

---

## Step 4: 范围锁定（防止蔓延）

每次必须明确回答以下 3 个问题，写入 spec 文件：

```markdown
## Scope Boundaries

### IN SCOPE（本次包含）
- [具体的文件/API/UI 元素]
- [具体的功能/行为]

### OUT OF SCOPE（明确排除）
- [不碰的模块]
- [不做的功能]
- [不处理的情况]

### HARD BOUNDARIES（硬边界）
- [绝对不能改的]
- [绝对不能依赖的]
```

**AI 会主动挑战模糊边界**：
- "你说的'优化性能'具体指什么指标？加载时间 <2s？API 响应 <200ms？"
- "'顺手改'的诱惑很大——但 X 不在本次范围，做吗？"

---

## Step 5: 增量草稿管理

每轮有意义的对话后，实时写入 `docs/spm/specs/draft-{topic-slug}.md`：

```markdown
# Draft: {Topic}

## 意图分类
- 类型: {新建/重构/...}
- 复杂度: {轻量/中等/复杂}

## 扫描结果
{Step 2 的发现}

## 测试决策
{Step 3 的结果}

## 范围定义
{Step 4 的 IN/OUT/HARD 边界}

## 待确认
- [ ] {待用户确认的点}

## 用户决策记录
- HH:MM: {用户说了什么}
```

**规则**：
- 第一篇回复就创建 draft
- 每次用户回应后更新
- 告知用户："草稿在 docs/spm/specs/draft-xxx.md，随时可以看"

---

## 访谈反模式（绝对禁止）

| ❌ 不要做的 | ✅ 应该做的 |
|-----------|-----------|
| 上来就写 Design Doc | 先分类、扫描、确认范围 |
| 问用户"你有没有 XX 库" | 自己扫，告诉用户"你有 XX" |
| "我建议..."（无事实依据） | "基于你的代码库里已有的 YY 模式，我建议..." |
| 一次抛出 5 个问题 | 每轮 1-2 个问题 |
| 在访谈阶段就拆 WBS | 访谈只定需求，WBS 是 Phase 2 的事 |

---

## 与 SPM 现有 Phase 1 的关系

```
原有流程:
  Soul-Searching (3问) → Brainstorming → Design Doc

增强后流程:
  ┌─────────────┐    ┌──────────────┐    ┌──────────────┐
  │ Step 1      │→  │ Step 2       │→  │ Step 3       │
  │ 意图分类    │    │ 研究前置     │    │ 测试评估     │
  └─────────────┘    └──────────────┘    └──────┬───────┘
                                                 │
  ┌─────────────┐    ┌──────────────┐    ┌──────▼───────┐
  │ Brainstorm  │←  │ Step 4       │←  │ Step 5       │
  │ +Design Doc │    │ 范围锁定     │    │ 增量草稿     │
  └─────────────┘    └──────────────┘    └──────────────┘
```
