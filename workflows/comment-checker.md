# Comment Checker — 去 AI 味注释审查

> 借鉴 OMO 的 Comment Checker。自动检测并标记 AI 风格的冗余注释，生成的代码像资深工程师写的。

## 触发条件

- Phase 4 质量门 / Code Review Stage 2 (Engineering Quality)
- 子代理提交代码后自动扫描

## 检测规则

### 🚫 应标记为冗余的注释模式

| 模式 | 示例 | 问题 |
|------|------|------|
| **功能复述** | `// This function calculates the total price` | 函数名已经说了 |
| **显而易见的操作** | `// Increment i by 1` 对应 `i++` | 代码自解释 |
| **TODO 但没行动** | `// TODO: optimize later` 没 ticket/日期 | 伪 TODO |
| **过度解释** | 10 行注释解释 3 行代码 | 信噪比失衡 |
| **模板残留** | `// Generated by...` `// Auto-generated` | AI 痕迹 |
| **废话摘要** | `// First, we check if x is null. Then we...` | 逐行翻译 |
| **情绪化注释** | `// This is a hack but it works` `// FIXME: terrible` | 模糊，不提供信息 |

### ✅ 应保留的注释

| 类型 | 示例 | 原因 |
|------|------|------|
| **业务规则** | `// Discount applied only to orders > $100 placed before noon` | 代码表达不了 |
| **非显而易见的算法选择** | `// Using Boyer-Moore here because input is >1MB typical` | 决策意图 |
| **已知限制** | `// Max 1000 items due to API rate limit (see INC-456)` | 链接到 ticket |
| **类型/接口文档** | JSDoc/TSDoc for public API | 契约定义 |
| **安全相关** | `// Deliberately not sanitizing — upstream guarantees clean input` | 安全决策 |

## 执行流程

```
1. 扫描本次提交中所有新增/修改的注释行
2. 对每行注释套用检测规则
3. 标记为 "redundant" / "keep" / "uncertain"
4. 生成报告（按文件分组）
5. 自动处理：
   - redundant ≥3 条 → 派发子代理清理（移除或改写）
   - redundant 1-2 条 → 标记在 Code Review 报告中，建议人工清理
   - uncertain → 列出但不强制处理
6. 清理后重新验证
```

## 报告模板

```markdown
## Comment Review Report — [commit/branch]

### Redundant (建议移除)
| 文件 | 行号 | 原文 | 问题类型 |
|------|------|------|---------|
| src/api.ts | 45 | `// Call the API endpoint` | 功能复述 |
| src/utils.ts | 12 | `// Auto-generated by assistant` | 模板残留 |

### Keep (保留，有价值)
| 文件 | 行号 | 原文 | 价值 |
|------|------|------|------|
| src/pricing.ts | 78 | `// Bus days only; see BA-234` | 业务规则 |

### Uncertain (请人工判断)
| 文件 | 行号 | 原文 |
|------|------|------|

### Summary
- 新增注释: 15 行
- 冗余: 3 (20%)
- 有保留价值: 8 (53%)
- 待定: 4 (27%)
```
