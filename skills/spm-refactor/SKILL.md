# SPM Refactor — 重构规范

## 自动检测

SPM Phase 3 的 TDD REFACTOR 阶段自动引用本技能。也可独立使用。

## 独立使用

不走 SPM 也可以直接说："按 spm-refactor 规范重构这个模块。"

---

## 重构铁律

```
1. 有测试才能重构
2. 每次只改一件事
3. 不改行为，只改结构
4. 改完立刻跑全量测试
```

---

## Agent 重构 6 大反模式

| # | 反模式 | 症状 | 为什么坏 |
|---|--------|------|---------|
| 1 | **"这个写得太复杂了，我简化一下"** | 删掉了原作者有意设计的状态机/策略模式 | 复杂是有原因的，简化=删功能 |
| 2 | **全文件重写** | 500 行文件全删，从头写 | 丢了所有边界处理和历史 bug 修复 |
| 3 | **"我顺便优化一下"** | 重构时顺手改业务逻辑 | 行为和结构同时变=没法定位回归原因 |
| 4 | **不跑测试就重构** | 重构完发现"咦怎么挂了" | 没有安全网，重构=赌博 |
| 5 | **跨文件大重构** | 一次改 10 个文件的"清理" | 比原始 PR 还大，reviewer 想杀人 |
| 6 | **引入新技术栈** | "这个 Redux 太重了，换成 Zustand" | 这不是重构，这是重写。需要单独立项 |

---

## 安全重构流程

```
1. 读测试 → 确认目前的行为被覆盖
2. 做一个小改动（重命名、提取函数、拆组件）
3. 跑测试 → 挂了吗？挂了就回退
4. 绿色 → commit
5. 重复 2-4，每次一个小改动
```

---

## 允许的重构操作

| 操作 | 示例 | 前提 |
|------|------|------|
| 重命名 | `getUserData` → `fetchUser` | 全局替换，测试绿 |
| 提取函数 | 20 行逻辑 → `calculateTotal()` | 测试覆盖这段逻辑 |
| 拆大文件 | 500 行 → `components/A.tsx` + `components/B.tsx` | 接口不变 |
| 去重 | 3 处相同逻辑 → 1 个 shared 函数 | 测试覆盖 3 处调用 |
| 改善命名 | `data` → `userList`, `temp` → `pendingOrders` | 只改名 |
| 删死代码 | 删掉注释掉的代码块 | 确认真的没人用 (grep) |

---

## 禁止的"重构"（需要单独 PR）

| 操作 | 为什么不是重构 |
|------|---------------|
| 换框架 | Redux → Zustand, Express → Fastify |
| 换语言/运行时 | JS → TS, REST → GraphQL |
| 改数据库 schema | 加列/改类型/换 ORM |
| 改业务逻辑 | "顺便修了那个奇怪的校验规则" |
| 改 API 接口 | 改字段名/状态码/响应格式 |

---

## 重构 commit 规范

```
refactor(auth): extract token validation to shared util

- 从 3 个路由提取 token 验证逻辑到 src/utils/validateToken.ts
- 行为不变，全量测试通过
- 无新增功能，无 bug 修复
```

**`refactor` commit 不应该包含任何行为变更。**
